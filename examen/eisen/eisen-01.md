# Opdracht 1: Plant werkzaamheden en bewaakt de voortgang

## Omschrijving

Je gaat in dit examenonderdeel bewijzen dat je de eisen en wensen van een klant kan omzetten naar user stories. Vervolgens dat je op basis van die user stories een globale planning kan maken en dat je tijdens je project de voortgang in de gaten kan houden met behulp van je planning.

### Waar kan je dit bewijs verzamelen?

- Stage  
- Deep Dives  
- Scrum Projecten in NexEd  
- Opdrachten voor externe partijen (bijv. Game Jams)

### Wat lever je op?

1. Project omschrijving (criterium 1.1)  
2. Backlog met stories (criterium 1.1 & 1.2)  
3. Planning (criterium 1.3)  
4. Bewijs van voortgangsbewaking (criterium 1.4)

---

## Criterium 1.1: Eisen/wensen user stories

| Criterium | 0 punten | 1 punt | 2 punten | 3 punten |
|---|---:|---:|---:|---:|
| 1.1. De eisen en wensen zijn verwerkt in de user stories. | De eisen en wensen van de opdrachtgever zijn niet verwerkt in de user stories. Er is geen of een summiere definition of done opgeleverd. | Enigszins: de eisen en wensen van de opdrachtgever komen beperkt terug in de user stories. | Grotendeels: uit de set van de user stories kun je opmaken dat de kandidaat de opdracht heeft begrepen en de essentiële zaken heeft verwerkt in de user stories. De definition of done is grotendeels compleet. | Volledig: de set van de user stories is een volledige weerspiegeling van de eisen en wensen, naast de essentiële eisen zijn ook de details verwerkt. Dit geldt ook voor de definition of done. |

### Randvoorwaarden

1. Er is een project omschrijving zodat terug te lezen is waar de stories op gebaseerd zijn.  
2. Er zijn minstens 5 user stories volledig uitgewerkt over 1 project.

### Veel gemaakte fouten

1. De eisen / wensen van de opdrachtgever zijn nergens terug te vinden.  
2. De student weet niet wat een Definition of Done is.  
3. Er is geen Definition of Done.  
4. De user stories volgen niet het format: ‘Als .. wil ik … zodat ik …’.  
5. De user stories hebben geen acceptatiecriteria.  
6. De user stories hebben geen prioriteit (bijvoorbeeld een MoSCoW analyse).

## Criterium 1.2: Criteria user stories

| Criterium | 0 punten | 1 punt | 2 punten | 3 punten |
|---|---:|---:|---:|---:|
| 1.2. De user stories voldoen aan de criteria (wie, wat, waarom en realistisch). | Niet of nauwelijks: de user stories zijn niet op de juiste wijze geformuleerd. | Enigszins: de user stories zijn op de juiste wijze geformuleerd, maar de prioritering en tijdsindicatie ontbreken of zijn niet realistisch. | Grotendeels: de user stories zijn op de juiste wijze geformuleerd en voorzien van prioritering en tijdsindicatie. De kandidaat kan in veel gevallen de user stories uitleggen en de prioritering en tijdsindicatie onderbouwen. | Volledig: de user stories zijn op de juiste wijze geformuleerd en voorzien van prioritering en tijdsindicatie. De kandidaat heeft een helder beeld bij de user stories en kan de prioritering en tijdsindicatie onderbouwen en plaatsen in de planning. |

### Veel gemaakte fouten

1. De user stories volgen niet het format: ‘Als .. wil ik … zodat ik …’.  
2. De stories hebben geen tijdsindicatie.  
3. De stories hebben geen prioriteit (via bijvoorbeeld een MoSCoW analyse).

## Criterium 1.3: Planning maken

| Criterium | 0 punten | 1 punt | 2 punten | 3 punten |
|---|---:|---:|---:|---:|
| 1.3. Op basis van de user stories is een complete en realistische planning gemaakt | Niet of nauwelijks: de planning is niet compleet en niet realistisch. Er ontbreken teveel user stories en het tijdspad is niet realistisch ingeschat. | Enigszins: de planning is enigszins compleet en realistisch. Er ontbreken nog user stories en het tijdspad is niet overal goed ingeschat. | Grotendeels: er is een planning. De planning is grotendeels compleet en de taken zijn doorgaans realistisch ingepland. | Volledig: er is een complete en realistische planning. De planning is compleet en realistisch ingeschat. |

### Veel gemaakte fouten

1. De user stories staan ergens in een markdown file maar er is geen aparte planning te vinden.  
2. De ureninschatting van de user stories is niet meegenomen, dus sprints hebben grote verschillen in de hoeveelheid werk.

### Voorbeelden

✅ **Goed**
- Een weekplanning met welke story verwacht wordt in welke week (of sprint). Dit kan in een markdown file beschreven worden.  
  ![Weekplanning](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/4a64865d-5f85-44f3-acb7-10611f2452a5/Untitled.png)
- Foto’s van je scrum board waarop de stories duidelijk zichtbaar zijn van minimaal 2 verschillende sprints.

❌ **Fout**
- CSV export uit je ticket systeem  
- Planning die niet met sprints is maar op basis van weken

## Criterium 1.4: Voortgang bewaken

| Criterium | 0 punten | 1 punt | 2 punten | 3 punten |
|---|---:|---:|---:|---:|
| 1.4. De voortgang is bewaakt en de juiste keuzes/afwegingen zijn gemaakt op basis van prioriteiten. | Niet of nauwelijks: de voortgang is niet bewaakt. Het is onduidelijk welke zaken wel of niet zijn afgerond. | Enigszins: er is een vorm van voortgangsbewaking. Men heeft beelden bij hetgeen is gemaakt en wat er nog moet worden gerealiseerd, maar onduidelijk is waar de prioriteiten liggen en hoe de voortgang past in de planning. | Grotendeels: er is een voortgangsbewaking. Er wordt gewerkt op basis van een tijdsplanning en prioritering. Er is echter een beperkt besef van keuzes en afwegingen wanneer zaken de planning verstoren. | Volledig: er is een duidelijke voortgangsbewaking op basis van prioritering. In de planning en prioritering is er zichtbaar sprake van keuzes en afwegingen. Dit niet alleen in het ontwikkelteam, maar ook met de opdrachtgever. |

### Veel gemaakte fouten

1. De voortgangsbewaking mist in het portfolio (dit gebeurt vaak).  
2. In de voortgangsbewaking is niets van ‘bewaking’ te zien. Bijvoorbeeld alleen screenshots van een backlog zonder verhaal — **je moet bewijzen dat je afwegingen / keuzes hebt gemaakt**.

### Voorbeelden

✅ **Goed**
- Binnen een sprint elke dag een screenshot maken van je scrum board en toelichting schrijven hoe je op basis van je planning (criterium 1.3) en je huidige voortgang wel/geen aanpassingen hebt gemaakt in planning/prioriteiten.

❌ **Fout**
- Screenshots van je scrum board zonder toelichting
