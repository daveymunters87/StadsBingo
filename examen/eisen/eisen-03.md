# Opdracht 3: Realiseert (onderdelen van) software

## 🎯 Doel
Je toont aan dat je (onderdelen van) software kunt **programmeren**, met **goede codekwaliteit**, **duidelijke structuur** en **versiebeheer via Git**.

---

## 📦 Wat lever je in
1. **Een complete repository**  
   (Frontend, backend en/of data-analyseproject dat werkt)
2. **Een overzicht van je Git-activiteit**  
   (Commits, branches, pull requests)

---

## 💻 Code – Criterium 3.1 t/m 3.5

| Nr. | Beoordelingsonderdeel | Voor 3 punten moet je laten zien dat... |
| --- | ---------------------- | --------------------------------------- |
| **3.1** | Realisatie van user stories | Alle toegewezen user stories zijn volledig gerealiseerd binnen de geplande tijd. |
| **3.2** | Functionaliteit voldoet aan eisen | Alle eisen en wensen uit de user stories zijn juist geïmplementeerd; de app werkt goed. |
| **3.3** | Codekwaliteit | De code is efficiënt, veilig, gestructureerd en bevat foutafhandeling en validatie (bijv. OOP, MVC). |
| **3.4** | Code conventions | Je houdt je consequent aan code conventions (zoals naamgeving, structuur, indentatie). |
| **3.5** | Leesbaarheid en commentaar | De code is netjes, logisch opgebouwd en voorzien van zinvol commentaar. |

---

## ⚙️ Randvoorwaarden code

Het project bevat minimaal:
- **Frontend/backend communicatie** (bijv. frontend → backend → database)  
- **Non-CRUD functionaliteit** (dus méér dan alleen standaard CRUD-acties)  
- **Minimaal 10 bestanden** met zelfgeschreven code (geen frameworkbestanden)
- **Data-analyseprojecten:** alle stappen van dataverwerking zijn aanwezig:  
  - Data verzamelen  
  - Data opschonen/transformeren  
  - Data interpreteren  
  - Data visualiseren (bijv. met Python of Power BI)

---

## ⚠️ Veelgemaakte fouten
- Incompleet project (niet uitvoerbaar door de examinator)  
- Geen instructies (README.md ontbreekt)  
- Code als afbeelding/pdf aangeleverd  
- Niet duidelijk welke code jij hebt geschreven  
- Te simpele of onveilige code (bijv. SQL-injectie mogelijk)

---

## 🧩 Versiebeheer – Criterium 3.6

| Nr. | Beoordelingsonderdeel | Voor 3 punten moet je laten zien dat... |
| --- | ---------------------- | --------------------------------------- |
| **3.6** | Gebruik van versiebeheer | Je gebruikt Git gestructureerd, met een duidelijke commitgeschiedenis, branches en pull requests. Alle versies zijn terug te halen. |

---

## ⚙️ Randvoorwaarden versiebeheer
- Minimaal **10 commits**  
- Minimaal **2 branches**  
- Er is gewerkt met **pull/merge requests**

---

## ⚠️ Veelgemaakte fouten
- Te weinig bewijs (bijv. slechts 1 commit screenshot)  
- Niet-informatieve commit messages  

---

## ✅ Voorbeelden van goed bewijs
- Link naar je GitHub/GitLab repository  
- Screenshot van je volledige commit history  

## ❌ Foute voorbeelden
- Screenshot van één commit  
- PDF met codefragmenten  
- Te weinig of onveilige code
